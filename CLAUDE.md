## What this repo is

Dependency Radar maps the dependency web across a folder of .NET git repositories and answers *"what needs retesting or redeploying if this project or NuGet package changes?"* via reverse-BFS blast-radius analysis. Full design rationale is in `DESIGN.md`; read it before making non-trivial changes.

## Commands

**Backend:**
```bash
dotnet build
dotnet test
dotnet run --project src/DependencyRadar.Service   # API on port 5000/5001
```

**Frontend** (separate runtime in local development):
```bash
npm install --prefix src/DependencyRadar.Web
npm run dev --prefix src/DependencyRadar.Web       # Vite dev server on port 5173
npm run build --prefix src/DependencyRadar.Web
npm run typecheck --prefix src/DependencyRadar.Web
```

**Docker:**
```bash
docker build -t dependency-radar .
docker run --rm -p 8080:8080 -v /path/to/repos:/repos dependency-radar
```

VS Code launch configs (`.vscode/launch.json`) support debugging backend, frontend, or both together. The fixture estate at `test/fixtures/` is the primary end-to-end smoke target.

The Docker image builds the frontend, publishes the backend, and serves the React app plus `/api` from `DependencyRadar.Service` on port `8080`.

## Architecture

Three components with a clean layering:

```
watched repo roots -> DependencyRadar.Service (watch + debounce + rescan) -> in-memory graph
                                                                            -> DependencyRadar.Web (React + Cytoscape)
```

**`DependencyRadar.Core`** — the library everything depends on. Handles filesystem walking with `.gitignore` pattern matching, `.sln` regex parsing, `.csproj` XML parsing, graph node/edge construction, project classification, and JSON serialization. No external dependencies and no network calls.

**`DependencyRadar.Service`** — ASP.NET Core minimal API. Watches configured roots (`DependencyRadar:Roots`), debounces bursts (default 1500 ms via `DependencyRadar:DebounceMilliseconds`), triggers a full rescan through Core (events invalidate, never mutate), and keeps the latest graph JSON in memory.

API surface: `GET /api/status`, `GET /api/graph`, `POST /api/rescan`, `GET /api/updates` (SSE).

`POST /api/rescan` is retained for diagnostics and development. The normal product workflow is watcher-driven rescans after local file changes.

**`DependencyRadar.Web`** — React 18 + TypeScript + Vite. Fetches the graph, renders it with Cytoscape (`fcose`, `dagre`, and `concentric` layouts), and computes blast radius, affected tests, affected deployments, consumers, dependencies, relationship route categories, and package-version tracing entirely client-side from the graph contract. The frontend should stay thin on domain logic — graph semantics are owned by Core and the API contract.

**The stable boundary** is the JSON shape produced by `DependencyRadar.Core` and served by `DependencyRadar.Service`. Frontend, tests, and Docker all consume that contract.

## Data model essentials

Nodes: `Repo`, `Solution`, `Project` (classified as `test | web | blazor | service | nuget-producing | library`), `Package`.

Edges: `solutionContains`, `projectRef`, `packageRef`, **`producedBy`** — the last one closes the internal NuGet loop (`Package → producing Project`) and is load-bearing for impact analysis. Package-reference edges can carry the locally observed `version` string.

Internal NuGet packages are consolidated onto the producing project in the UI. The project is the primary graph and Impact Panel target; internal package IDs act as search aliases and version carriers rather than duplicate primary nodes.

Stable node IDs are generated by `GraphBuilder.IdFor(kind, raw)`.

## Current frontend behaviour

- Search opens on focus and shows all repo, project, and external package suggestions before typing.
- The default graph layout is **Cluster Map** (`fcose`). **Dependency Paths** and **Most Referenced** remain available.
- The **Repositories** toggle shows or hides repo grouping boxes. Selecting a repo turns repo grouping back on.
- The selected-node side surface is the **Impact Panel**. It shows Affected Tests, Affected Deployments, All Consumers, All Dependencies, and External packages.
- Impact Panel sections are collapsible. Affected Tests and Affected Deployments are expanded by default.
- Relationship badges are **Direct Project**, **Direct Package**, and **Indirect Package**.
- Version pills show consumed NuGet package versions from local `PackageReference` edges. Tooltips explain which project is using which package version.
- Project type filters also filter Impact Panel project lists.
- Graph control tooltips render through a portal so dock/panel overflow does not clip them.

## Locked design decisions

1. **No metadata fetching** — never call nuget.org, GitHub, or any network endpoint from the scanner.
2. **Unknowns are first-class** — unresolvable references stay `unknown`, never guessed.
3. **Multi-TFM = one node** — dependencies are unioned across target frameworks.
4. **File watcher events trigger full rescan** — not incremental graph mutation.
5. **`producedBy` edge is load-bearing** — do not remove or rename it without updating impact analysis.
6. **Parsers avoid `Microsoft.Build`** — `.sln` via regex, `.csproj` via XML.
7. **Internal NuGet packages are consolidated in the UI** — preserve producer-project identity while keeping consumed version tracing visible.

## Conventions

- `DependencyRadar.Core` and `DependencyRadar.Service` compile with `TreatWarningsAsErrors=true` and `Nullable=enable`.
- Types are `internal` by default; `InternalsVisibleTo("DependencyRadar.Tests")` exposes scanner internals to tests.
- When adding new edge kinds: update `EdgeKind`, JSON serialization, and frontend rendering together.
- When adding new project classifications: update the classifier, JSON shape, and frontend filters/styles together.
- For parser edge cases, add targeted parser unit tests rather than relying solely on fixture runs.
- User-facing copy should say **NuGet package**, not "packable"; keep the JSON key `nuget-producing` stable.
- Display name prefix stripping is configured through `DependencyRadar:NamePrefixes` and must not alter stable IDs or paths used for file IO.
